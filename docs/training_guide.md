# AnyUp3D Training Guide

## How Training Works

AnyUp3D is a **self-supervised video feature upsampler**. It learns to take low-resolution video features and produce high-resolution ones — guided by the raw RGB frames — without ever needing human annotations or segmentation masks.

### The Big Picture

```
Low-res video clip  ──►  AnyUp3D model  ──►  Predicted high-res features
                                                        │
                                              compare (loss)
                                                        │
High-res video crop ──►  frozen VideoMAE  ──►  GT high-res features
```

The supervision signal comes entirely from a **frozen pretrained VideoMAE** encoder. The model is never told what objects are in the scene — it simply learns to match what VideoMAE would extract from a high-resolution crop.

---

## The Training Loop Step by Step

### 1. Data Loading

Each batch contains three things from `VideoDataset`:

| Key | Shape | Description |
|-----|-------|-------------|
| `video` | `(B, C, T, H, W)` | Low-res input clip, values in `[0, 1]` |
| `video_crop` | `(B, C, T, Hc, Wc)` | High-res crop of the same clip |
| `patch_size` | `int` | Spatial patch size for this batch |

No masks, no labels, no annotations of any kind.

### 2. GT Feature Extraction (frozen, no gradient)

`video_crop` is passed through a **frozen VideoMAE-Base** encoder. Its output — a spatiotemporal feature volume `(B, D, T', H', W')` where `D=768` — serves as the ground truth that AnyUp3D is trained to reproduce.

VideoMAE is never updated during training.

### 3. Forward Pass

`video` (low-res) and `gt_features` are fed into AnyUp3D. The model outputs `pred_features` at the same resolution as the GT feature volume.

### 4. Loss Computation

Training minimises four losses combined as a weighted sum:

```
L_total = L_recon
        + λ1 · L_input
        + λ2 · L_self
        + λ3(t) · L_temporal
```

#### `L_recon` — Reconstruction Loss
Cosine similarity + MSE between predicted features and VideoMAE GT features. This is the primary learning signal.

#### `L_input` — Input Consistency
The predicted high-res features, when downsampled back to low resolution, should match the coarse input features. Prevents the model from hallucinating structure that wasn't there.

#### `L_self` — Self-Consistency
The model's output should be stable under photometric augmentations (brightness, contrast, Gaussian noise) applied to the guidance video. The augmented branch runs under `no_grad` and acts as a pseudo-target.

#### `L_temporal` — Temporal Consistency
Adjacent frame features should not flicker. This loss is **RGB-gated**: pairs where the scene changes significantly (large RGB difference between frames) are excluded from the penalty. No masks needed — the RGB signal itself tells the loss where motion occurred.

`λ3` is linearly warmed up from 0 over the first `temporal_lambda_warmup_steps` steps (default: first 1000 steps). It stays at 0 during the T=1 warmup stage since there are no adjacent frame pairs.

### 5. T-Curriculum Scheduler

Training starts with short clips (`T=1`) and gradually increases the number of frames as training progresses. This is configured in `t_schedule` in the config file:

```yaml
t_schedule:
  - {t: 1, start_step: 0,    batch_size: 8}
  - {t: 4, start_step: 5000, batch_size: 4}
  - {t: 8, start_step: 15000, batch_size: 2}
```

As `T` increases, batch size decreases to keep memory usage roughly constant. The DataLoader is rebuilt whenever a stage transition occurs.

### 6. Checkpointing

- **Periodic checkpoints** are saved every `save_every_n_steps` steps as `step_XXXXXX.pth`.
- **Best checkpoint** (`best.pth`) is saved whenever the validation metric improves.
- Checkpoints store: model weights, optimizer state, current step, and current T-stage — enough to resume exactly.

---

## What Happens with Segmentation Masks?

**They are not used during training.** The training loop is mask-free by design.

Masks only appear during **post-training evaluation** (`eval_linear_probe.py`), where a linear probe is trained on frozen upsampled features and evaluated for semantic segmentation (mIoU) on ADE20K or Cityscapes. That is a separate step entirely, not part of training.

## Choosing a Dataset

Because training only needs raw video frames — no annotations — your dataset choice comes down to **size, diversity, and availability**.

### What the pipeline requires

- Video files in any of: `.mp4`, `.avi`, `.mov`, `.mkv`, `.webm`
- At least **16 frames per video** (shorter clips are filtered out automatically)
- A flat or nested directory structure — `VideoDataset` scans recursively
- A `train.txt` / `val.txt` manifest generated by `prepare_val_split.py`

No class labels, no segmentation masks, no optical flow, no captions.


---

### Dataset Comparison

| Dataset | Size | Videos | Good for AnyUp? | Notes |
|---------|------|--------|-----------------|-------|
| **Kinetics-400** | ~240 GB | ~240k clips | **Best choice** | Large, diverse, standard benchmark |
| **Kinetics-700** | ~500 GB | ~540k clips | **Best choice** | Even more diverse |
| **Something-Something v2** | ~20 GB | ~220k clips | Good | Strong temporal diversity; hand-object interactions |
| **UCF-101** | ~7 GB | ~13k clips | Acceptable | Decent size; action-focused; easy to download |
| **HMDB-51** | ~2 GB | ~7k clips | Minimal | Very small; only for quick smoke tests |
| **YouTube-VOS** | ~17 GB | ~4.5k videos | Acceptable | Segmentation dataset; annotations unused; decent variety |
| **DAVIS 2017** | ~2.5 GB | ~150 clips | Testing only | Too small for real training; fine for pipeline validation |

---

### Why DAVIS and YouTube-VOS are not ideal for training

Both DAVIS 2017 and YouTube-VOS are **segmentation datasets** — their videos are short, carefully selected object-tracking sequences. They work with the pipeline because masks are simply ignored, but:

- DAVIS has only ~150 clips total — far too few to train a model that generalises.
- YouTube-VOS has ~4.5k videos, which is borderline but still domain-biased toward salient foreground objects.
- Neither provides the natural scene diversity that makes VideoMAE features meaningful.

Use them to **verify the pipeline** end-to-end (generate manifests, run a few steps, confirm loss goes down), then switch to a proper large-scale dataset for real training.

### Recommended workflow

#### Step 1 — Download a dataset

```bash
# Example: UCF-101 (small, fast to download)
# Download from: https://www.crcv.ucf.edu/data/UCF101.php
# Extract to data/videos/

# Example: Kinetics-400 (recommended for real training)
# Use the official downloader or a mirror
```

#### Step 2 — Generate manifests

```bash
python anyup/scripts/prepare_val_split.py \
    --video_dir data/videos/ \
    --output_dir configs/ \
    --val_fraction 0.05 \
    --seed 42
```

This writes `configs/train.txt` and `configs/val.txt`. The split is deterministic — running it again with the same seed always produces the same files.

#### Step 3 — Run the test suite to validate your setup

```bash
pytest tests/test_val_dataset.py --manifest_dir configs/ -v
```

All four real-data tests should now pass (no more skips).

#### Step 4 — Start training

```bash
python train_3d.py --config configs/anyup3d_train.yaml

# For a quick smoke test (10 steps, T=1, batch=1):
python train_3d.py --config configs/anyup3d_train.yaml --debug
```

---

## Post-Training Evaluation

After training, evaluate feature quality with a segmentation linear probe:

```bash
# Requires ADE20K or Cityscapes (separate download)
python anyup/scripts/eval_linear_probe.py \
    --checkpoint checkpoints/best.pth \
    --dataset ade20k \
    --data_dir /data/ade20k/
```

This is the step where segmentation masks are finally used — but as evaluation targets for the linear probe, not as training supervision.
